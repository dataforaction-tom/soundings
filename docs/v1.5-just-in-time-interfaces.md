# Soundings v1.5 — Just-in-time interfaces

**Status:** Draft 0.1
**Scope:** Richer interfaces over the v1 orchestration layer. No new sources, no new fundamental contracts — just better ways to make sense of what's already there.
**Builds on:** [v1](./v1-orchestration-and-capture.md)
**Related:** [README](../README.md) · [v2](./v2-context-layer.md) · [v3](./v3-contribution-layer.md)

---

## 1. Purpose

v1 ships a working MCP server and a thin UI. v1.5 adds three substantive interface capabilities that make the data actually feel like an insight commons rather than a data lookup tool:

1. **Server-side narrative briefs** — a tool that produces written, sourced summaries about a place or a question, designed for humans to read.
2. **Just-in-time dashboards** — Glimpse-style on-demand visual views, generated from a question rather than pre-built.
3. **The public `/asks` view** — a visual surface for the questions corpus, showing what's being asked, where, and what the gaps are.

None of these change the v1 tool contracts. They sit alongside them.

---

## 2. Design intent

The shift from v1 to v1.5 is the shift from "a place to look things up" to "a place to think with". v1 answers what's there. v1.5 answers what it means and what people are asking.

This is also where Soundings starts paying back its capture layer. The questions corpus has been growing since v1 day one. v1.5 makes it visible.

## 3. New tools

### 3.1 `get_narrative_brief`

A server-side tool that produces a written brief about a place or topic. Wraps several lower-level tools and uses an LLM to compose the result, with strict citation requirements.

```yaml
input:
  question: string             # natural language
  place_id: string | optional
  topics: [string] | optional  # ["deprivation", "health"]
  length: "short" | "medium" | "long" | optional
  audience: "general" | "funder" | "commissioner" | optional
output:
  brief:
    title: string
    summary: string             # markdown
    sections:
      - heading: string
        markdown: string
        sources_used: [string]  # source_ids cited in this section
    place_id: string | optional
    period_covered: string
    sources: [SourceRef]        # full list of sources cited
    confidence_note: string     # honest uncertainty statement
    questions_we_couldnt_answer: [string]  # surfaced as gaps
    generated_at: ISO8601
```

The brief generator is a small agent loop: it plans a set of underlying tool calls, executes them, then composes a brief from the results. Citations are enforced at composition time — the LLM is constrained to only reference values that appeared in tool results.

The `questions_we_couldnt_answer` field is deliberate: when the brief composer wanted a number that wasn't available, that's logged as a gap and surfaced to the user. These gaps also feed the corpus, so the public `/asks` view can show them.

**Model choice.** v1.5 uses a configurable LLM backend, with the local Mac Studio (when available) as the default and Anthropic API as fallback. Briefs are not real-time-critical; some can be queued.

### 3.2 `compose_dashboard`

Just-in-time dashboard generation. Borrows directly from Glimpse.

```yaml
input:
  question: string             # "How is housing affordability changing in the North East?"
  place_id: string | optional
  preferred_format: "auto" | "comparison" | "trend" | "profile" | optional
output:
  dashboard:
    id: uuid                   # for sharing / re-rendering
    title: string
    description: string
    panels:
      - type: "indicator" | "trend" | "comparison" | "table" | "narrative"
        title: string
        spec: object           # panel-type-specific
        sources: [SourceRef]
    questions_we_couldnt_answer: [string]
    generated_at: ISO8601
    permalink: string          # /dash/{id}
```

The composer chooses a layout from a small set of panel types based on the question shape, fills each panel from underlying tool calls, and persists the result. Dashboards are immutable once composed (re-asking produces a new one) but can be referenced by permalink.

**No bespoke charting library in v1.5.** Panels render with Observable Plot or similar — proven, accessible, and lightweight. Charting sophistication is not where the value is; question understanding is.

### 3.3 `find_similar_questions`

Surface previously-asked questions similar to a new one, from the corpus. Useful both as a UI feature ("others have asked this") and as a feedback loop into the brief composer (look at how related questions were answered).

```yaml
input:
  question: string
  place_id: string | optional
  limit: integer | optional
output:
  similar:
    - question_id: uuid
      question: string         # only if originally captured at full consent
      tool_path: [string]      # which tools answered it
      asked_at: ISO8601
      asker_sector: string | optional
      result_status: string
```

This is the first user-facing read from the corpus. It only ever surfaces questions captured at full consent.

## 4. The public `/asks` view

A new section of the reference UI exposing the questions corpus visually.

**`/asks`** — landing page. Shows aggregate patterns: top question types this month, geographic heatmap of activity (at LTLA resolution, never finer), most-cited sources, and a "questions we couldn't fully answer" feed.

**`/asks/places/{id}`** — what's being asked about this place. The corpus filtered to a single geography, with consent-respecting redaction.

**`/asks/topics/{topic}`** — what's being asked about a particular topic across all places.

**`/asks/gaps`** — the gap analysis view. Questions the orchestration layer couldn't answer well, sorted by frequency. This is the most strategically useful view: it tells funders and data publishers what's missing.

All `/asks` views run from the published, sanitised corpus. No raw store data ever appears here.

## 5. UI changes from v1

The v1 UI is extended, not replaced.

- **`/`** — the home text box now offers a choice after submission: "Read a brief" (uses `get_narrative_brief`), "See a dashboard" (uses `compose_dashboard`), or "Just the data" (the original v1 path).
- **`/place/{id}`** — gains a "Generate brief about this place" button, and a "Recently asked about this place" panel pulling from the corpus.
- **`/dash/{id}`** — new. Persistent dashboard pages, shareable.
- **`/asks`** and children — new, as above.

The UI remains server-rendered Astro. Dashboards are static once composed; only the home/composer pages need any client-side interactivity.

## 6. Caching and rate-limiting

Brief generation and dashboard composition are expensive. v1.5 introduces:

- **Result caching.** Briefs and dashboards are content-addressed by the question + relevant tool inputs. Identical questions within a freshness window return the cached result.
- **Per-session rate limits.** Anonymous users get a generous-but-finite budget per day. Authenticated organisations (introduced in v2) get a higher one.
- **Async generation.** Long-running briefs can be queued; the UI shows a progress state and emails or notifies when ready (notification mechanism deferred to v2, where identity exists).

## 7. New corpus fields

The `QuestionRecord` from v1 gains optional fields in v1.5. These are additive — older records remain valid.

```yaml
QuestionRecord:
  # ... all v1 fields ...

  # v1.5 additions
  composed_artefact:
    type: "brief" | "dashboard" | null
    artefact_id: uuid | null
    panels_or_sections_count: integer | null

  gap_signals: [string]        # specific things the composer wanted but couldn't get

  derived_from_question_id: uuid | null   # if generated via "ask similar" workflow
```

The `gap_signals` field is the corpus's most actionable artefact: a structured stream of things people are asking about that the data doesn't yet support. This becomes the backlog for v2 and v3.

---

## 8. Out of scope for v1.5

- Authentication / user accounts (introduced in v2 alongside the context layer).
- Editing or annotating briefs and dashboards (read-only in v1.5).
- Custom dashboard composition beyond the built-in panel types.
- Embedding briefs/dashboards in third-party sites (worth considering for v1.6).
- Any new upstream data sources beyond the v1 set.
- Anything from the contribution layer (v3) — briefs in v1.5 are still composed entirely from official statistics.

---

## 9. Open questions

1. **Cost ceiling.** Brief generation has a real cost when running against API models, even with local fallback. What's the ceiling per anonymous session, and what does it imply about funding?
2. **Brief evaluation.** How do we know a brief is good? At minimum, a thumbs-up/down feedback mechanism captured into the corpus. More structured evaluation is worth designing into v1.5 from the start.
3. **Public dashboard discoverability.** Should permalinked dashboards be indexed and browsable, or only accessible by direct link? Privacy implications either way.
